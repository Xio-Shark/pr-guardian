# PR Guardian｜规则覆盖与验证清单

## 1. 一眼能看到的数字

- `5` 条内置规则
- `3` 个 GitHub 输出通道
- `4` 个 LLM Provider
- `73` 个测试函数
- `1` 套结构化 Finding 模型

这些数字说明它不是“只会扫一类问题”的单点工具，而是一个有规则层、模型层、回写层和回归测试的完整审查链路。

## 2. 规则覆盖清单

当前内置规则文件共 `5` 个：

- `secrets_scan.py`
- `lockfile_consistency.py`
- `affected_tests.py`
- `min_permissions.py`
- `changelog_breaking.py`

对应覆盖的风险面：

- 密钥泄露
- 依赖清单与 lockfile 不一致
- 受影响测试缺失
- GitHub Actions 权限过宽
- Breaking Change 缺少变更说明

## 3. 输出通道

Reporter 当前固定输出 `3` 类 GitHub 结果：

- `Check Run`
- `Review Comments`
- `Summary Comment`

这里不是“想发什么就发什么”，而是先把 Findings 结构化，再稳定映射到三个通道。

## 4. 模型适配面

当前仓库里保留 `4` 个 Provider 适配：

- `openai`
- `glm`
- `minimax`
- `kimi`

这让项目的重点从“会不会调某个模型接口”，转到“结构化审查输出是否稳定”。

## 5. 测试面

当前 `tests/` 下可统计到 `73` 个测试函数，重点覆盖：

- 单条规则行为
- Reporter 三通道输出
- policy 合并与默认值
- context builder
- GitHub API 适配层

这也是为什么 README 里可以更有底气地写 `pytest --cov=src/pr_guardian`。

## 6. 为什么这些数字重要

- `5` 条规则说明它已经覆盖多个高确定性审查面
- `3` 个输出通道说明它不只是本地打印 findings
- `73` 个测试函数说明规则和 reporter 都有回归基础
- `4` 个 provider 说明模型层是抽象能力，而不是单一供应商绑定
