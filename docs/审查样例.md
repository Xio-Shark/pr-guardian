# PR Guardian 审查样例

本文档用仓库中的规则测试样例，展示这个项目如何把“PR diff -> Finding -> 审查输出”做成结构化流水线。

## 样例 1：新增 AWS Access Key，会触发密钥扫描

### 输入 diff

来自 `tests/test_secrets_scan_rule.py`：

```diff
@@ -1,1 +1,1 @@
+aws_key = "AKIA1234567890ABCDEF"
```

### 规则输出

根据 `secrets_scan.py` 的规则实现，这一行会命中 `aws_access_key` 模式，并生成 `error` 级 Finding：

```json
{
  "rule_id": "security/secrets-scan",
  "severity": "error",
  "message": "检测到疑似硬编码密钥 (aws_access_key)。",
  "evidence": [
    {
      "file": "src/auth.py",
      "line": 1
    }
  ]
}
```

### 这个样例说明了什么

- 工具只扫描新增行，不会把删除行误报成问题
- 输出是结构化 Finding，不是模糊文本
- 审查结果可以直接进入 Check Run 或 Review Comment

## 样例 2：改了依赖清单但没改 lockfile，会被阻断

### 输入 diff

来自 `tests/test_lockfile_consistency_rule.py`：

```diff
@@ -1 +1 @@
-old
+new
```

文件路径：

```text
package.json
```

### 规则输出

当 `package.json` 变更但对应 lockfile 没变更时，规则会输出：

```json
{
  "rule_id": "deps/lockfile-consistency",
  "severity": "error",
  "message": "检测到 `package.json` 变更，但未发现对应 lockfile 变更。预期之一: package-lock.json, yarn.lock, pnpm-lock.yaml"
}
```

### 这个样例说明了什么

- 这个项目不只是做“安全扫描”，也会做工程质量治理
- 依赖清单和 lockfile 的一致性被显式纳入规则层
- 规则输出能直接用于阻断 PR 合并

## 样例 3：GitHub Actions 权限过宽，会被定位到具体行

### 输入 diff

来自 `tests/test_min_permissions_rule.py`：

```diff
@@ -0,0 +1,6 @@
+name: CI
+on: [push]
+permissions: write-all
+jobs:
+  build:
+    runs-on: ubuntu-latest
```

### 规则输出

```json
{
  "rule_id": "ci/min-permissions",
  "severity": "error",
  "message": "workflow 使用 `permissions: write-all`，权限过宽。",
  "evidence": [
    {
      "file": ".github/workflows/ci.yml",
      "line": 3,
      "snippet": "permissions: write-all"
    }
  ]
}
```

### 这个样例说明了什么

- 工具能把问题定位到具体文件和具体行
- 审查不仅关注应用代码，也覆盖 CI 安全配置
- “最小权限”这种工程规范被固化成机器可执行规则

## 为什么这不是“纯规则”工具

这个项目的主线不是只写规则，而是把审查流程分成两层：

1. **确定性规则层**  
   负责密钥扫描、lockfile 一致性、CI 权限、breaking change 等稳定规则。
2. **LLM 审查层**  
   只在需要时运行，且必须输出结构化 JSON 和证据，避免生成不可控结论。

这意味着它既保留了规则的稳定性，也保留了模型对上下文的理解能力。

## 小结

本项目的核心设计不是”把大模型接进代码审查”，而是：

- 先规则、后模型的分层流水线
- 结构化 Finding 输出
- 证据回溯
- 可直接接入 GitHub 三通道输出
