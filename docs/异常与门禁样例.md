# PR Guardian｜异常与门禁样例

## 1. 说明

这页只看坏情况：

- 什么情况下会直接卡门禁
- 什么情况下不会重复刷评论
- 模型层出错时会不会把结果搞坏

## 2. 只要存在 error 级 Finding，Check Run 就会失败

场景来源：`tests/test_github_reporter.py`

已验证行为：

- 构造 `60` 条 `error` 级 Finding
- `GitHubReporter._publish_check_run(...)` 返回：
  - `conclusion = failure`
  - `annotation_count = 50`

这说明：

- `error` 级 Finding 会直接进入 gating
- annotation 会被截断到 `50` 条，避免过量输出把 GitHub 通道刷爆

## 3. 历史评论命中相同 fingerprint 时，不会重复创建行内评论

场景来源：`tests/test_github_reporter.py`

已验证行为：

- 历史评论 body 中已包含同一条 finding 的 fingerprint
- 再次调用 `_publish_review_comments(...)`
- 返回 `comment_count = 0`
- `review_calls == []`

这说明：

- reporter 不是“每次重跑都再发一遍”
- 同一问题会被去重，而不是反复骚扰 reviewer

## 4. 策略文件配置错误会被显式指出

场景来源：`tests/test_policy.py`

已验证行为包括：

- `llm.provider` 填未知值时，校验错误会指出 provider 不支持
- `severity_overrides` 覆盖未启用规则时，会返回配置错误
- `include` 与 `exclude` 模式冲突时，会返回冲突错误

这说明：

- 项目不是“配置随便写，运行时再炸”
- policy 层本身也有有效性检查

## 5. 模型输出结构不合法时，会转成结构化 error Finding

场景来源：`tests/test_llm_providers.py`

已验证行为：

- Provider 返回了不符合 schema 的 JSON
- `generate_structured(...)` 返回：
  - `parsed is None`
  - `findings[0].rule_id == "llm/schema-validation"`

这说明：

- 模型层不是直接把坏 JSON 往下游传
- 结构错误也会被当成受控 Finding 处理

## 6. Provider 鉴权失败时，会统一归一成 provider-error

场景来源：`tests/test_llm_providers.py`

已验证行为：

- Kimi provider 返回 `401`
- `generate_structured(...)` 返回：
  - `parsed is None`
  - `findings[0].rule_id == "llm/provider-error"`
  - `findings[0].severity == error`

这说明：

- 外部模型调用失败不会让整个流程失控
- provider 失败被归一化成统一错误类型，便于后续回归和告警
